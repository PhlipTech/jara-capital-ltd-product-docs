4‑month delivery timeline (8 sprints)

JARA CAPITAL LIMITED
Scope: MVP (Mobile + Backend + Admin) · Cadence: 1 month = 2 sprints (2‑week sprints) · Output: milestones + sprint plan
Month 1: full consumer onboarding (eKYC via PhlipVerify per PRD — use the same sprint breakdown if your vendor is FlipCheck or another provider), forgot password, PIN, and admin frontend kickoff.
Month 2: RBAC (roles/permissions), staff admin login + invite/add other admin users, customer/user module; savings packages and investment packages created and managed in admin (not sourced from Cowrywise catalog); mobile product screens read the app catalog (admin-defined).

Schedule risk note
Product catalog is admin-authored in-app (savings and investment packages). Cowrywise (or embed partner) remains for accounts, wallet, settlement after Month 2. Critical path from Month 3 onward: linkage → wallet / VA → funding → execute invest (mapping admin products to partner instruments as per technical design) → portfolio / activity → withdraw. PRD open items can shift Months 3–4.

Milestones

M1: Onboarding + KYC + admin kickoff

End of Month 1

Full mobile onboarding (register, forgot password, PIN, terms), PhlipVerify eKYC integrated end-to-end, outcomes stored; admin web app scaffolded with auth shell and first routes in staging.

M2: RBAC, staff provisioning, admin catalog, mobile product flow

End of Month 2

Admin: role-based access control, staff login, invite/add users and assign roles; customer directory/detail; create and manage savings packages and investment packages (app-owned catalog). Mobile: savings & investment product list/detail/filters/disclosures served from your catalog API (not Cowrywise product list).

M3: Money movement (linkage, wallet, invest)

End of Month 3

Cowrywise linkage live; users can fund wallet and complete investments with idempotency, retries, and admin investment tracking.

M4: Portfolio, withdraw, UAT, go‑live

End of Month 4

Portfolio and activity complete; withdraw where supported; compliance hardening done; full UAT and production release executed.

Sprint plan

Editable schedule
Sprint Theme Key deliverables Exit gates
Sprint 1 (Weeks 1–2)
Month 1 · Weeks 1–2
 Onboarding backend + mobile auth + admin frontend kickoff
  • Backend: app user model, sessions, consent records, secure PIN storage (hashed), forgot-password flow (email/OTP per policy)
  • Mobile: register, login, logout, forgot password, create/confirm PIN, terms & partner consent, basic profile
  • Backend: PhlipVerify server-side session/init endpoints; store correlation IDs (no secrets on device)
  • Admin frontend: project scaffold, routing, layout shell, login screen wired to staging API (or mock), environment config
  • Webhook receiver skeleton (signature + raw payload audit) for PhlipVerify callbacks
  • Happy-path onboarding works in staging (up to KYC handoff)
  • PIN and password flows meet security baseline (Keychain/Keystore, rate limits)
  • Admin app builds and loads authenticated shell
Sprint 2 (Weeks 3–4)
Month 1 · Weeks 3–4
 Full KYC (PhlipVerify) + onboarding completion + admin foundations
  • Mobile: complete eKYC UX (liveness, BVN, NIN, address per enabled modules), blocked/retry/partial states, profile KYC status
  • Backend: PhlipVerify webhooks idempotent; persist per-check and overall status; NDPR-aligned minimisation
  • Backend: onboarding completion rules (what “verified” means); block downstream money actions until policy satisfied
  • Admin frontend: staff login/session and navigation (light route protection only; full RBAC matrix lands in Month 2)
  • Admin backend (parallel, optional): stub staff table or auth provider hook-up only if needed for KYC ops demos
  • KYC outcomes accurate; no false “verified” if required checks fail
  • Duplicate webhook events do not corrupt state
  • Admin can authenticate; role matrix deferred to Month 2 except basic “logged in vs not”
Sprint 3 (Weeks 5–6)
Month 2 · Weeks 5–6
 RBAC + staff users + customer module + admin-created savings & investment packages
  • RBAC: roles, permission matrix, server-side enforcement on all admin APIs; audit log for permission-sensitive actions
  • Staff: admin login, session management; invite/create other staff users; assign roles; activate/deactivate/suspend; password reset (per policy)
  • Admin: customer/user directory (search, filters: onboarded, KYC state), read-only customer detail (profile + KYC summary)
  • Admin + backend: create, edit, list savings packages and investment packages (app database — e.g. name, type, currency, min amount, tenor, risk/disclosure fields, status draft/active)
  • Backend: public/catalog API for mobile — returns only active, visible packages (no Cowrywise product-list dependency)
  • Mobile: Invest tab — product list + detail wired to app catalog API; empty/error/offline states
  • Unauthorized roles cannot access restricted admin actions (verified by tests)
  • Privileged staff can onboard another admin and assign a role
  • Created packages appear on mobile when marked active/visible
Sprint 4 (Weeks 7–8)
Month 2 · Weeks 7–8
 Package lifecycle + compliance fields + customer depth + mobile journey polish
  • Admin: package workflow — draft → review → active → archived; visibility in app, featured, display order, compliance/disclosure copy
  • Admin: user detail — expand with internal notes, KYC timeline, and (where available) linkage/wallet summary placeholders for Month 3
  • Backend: validation rules for packages (min/max, currency, status transitions); enforce visibility on catalog API
  • Mobile: filters, category tabs (savings vs investment), disclosures and “not ready to invest” vs ready states; optional partner product code field stored admin-side for later Cowrywise mapping (Month 3)
  • Package changes affect mobile catalog without redeploy
  • Support can explain customer KYC + which packages exist from admin
  • Audit trail covers package and RBAC changes
Sprint 5 (Weeks 9–10)
Month 3 · Weeks 9–10
 Cowrywise linkage + wallet + funding
  • Backend: create/link Cowrywise account after KYC policy satisfied; persist customer + wallet identifiers
  • Mobile: wallet balance, virtual account details, funding instructions, balance refresh (poll/webhook)
  • Admin: per-user and aggregate wallet balance views with as-of timestamp / stale indicator
  • Outbound payment reliability foundations for later invest (idempotency store, audit events)
  • User can fund wallet via VA and see updated balance within agreed delay
  • Linkage and wallet data consistent between mobile and admin read models
Sprint 6 (Weeks 11–12)
Month 3 · Weeks 11–12
 Execute invest + reliability + admin investment operations
  • Mobile: invest flow — amount validation, confirm, submit; success/failure UX
  • Backend: execute investment via Cowrywise using mapping from admin-defined package to partner product/fund identifier (stored on package or config); idempotency keys; retries with backoff/jitter; DLQ + alerts
  • Admin: investment requests queue, request detail, progress timeline, correlation to user and product
  • Observability: structured logs, metrics, tracing, error-rate and provider latency alerts
  • No double-invest on duplicate submits
  • Admin can trace any investment request end-to-end
Sprint 7 (Weeks 13–14)
Month 4 · Weeks 13–14
 Portfolio + activity + withdraw + compliance hardening
  • Mobile: portfolio positions and performance summary; activity feed with filters; transaction detail
  • Mobile: withdraw/redeem where API supports; honest SLA copy
  • Admin: global and per-user transaction history (filters, pagination, export hooks)
  • Privacy notice + consent versioning hardening (NDPR); penetration test fixes as needed
  • Statuses consistent across mobile and admin
  • Withdraw only offered when product rules allow
Sprint 8 (Weeks 15–16)
Month 4 · Weeks 15–16
 UAT + release prep + go-live
  • Full E2E regression: onboarding + KYC + catalog + fund + invest + portfolio + withdraw
  • Admin regression: staff, users, products, investments, balances, audit exports
  • Production readiness: backups, monitoring, on-call, rollback plan; go/no-go sign-off
  • No open P0/P1 defects; runbooks finalized
  • Go-live sign-off recorded
